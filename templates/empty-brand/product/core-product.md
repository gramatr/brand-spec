# Core Product

> **This is a sample file.** Replace with your brand's real product
> description. One file per product or product-line. Single-product
> brands keep one file; multi-product brands add a file per product.
> Delete this callout when you replace the content.

## What it is

A one-paragraph mechanical description. What does the product *do*, in
plain language, with no marketing varnish. A new engineer or a new
salesperson should be able to read this and not be misled.

Acme is a {category} that {core job}. Customers use it to {primary
outcome} by {primary mechanism}.

## How it works

The honest mechanical loop. If a feature has limits, name them here so
downstream copy doesn't overpromise.

1. The user {does first thing} via {interface}.
2. The system {processes / transforms / routes} the input.
3. The user receives {output} within {time bound}.

## What's included

- Capability A — short description
- Capability B — short description
- Capability C — short description

## What's NOT included

Be explicit. The "what we don't do" list is a feature for buyers and
prevents misalignment.

- Not a {adjacent thing}. We integrate with those, we don't replace them.
- Not available in {region/segment} as of this writing.

## Pricing model

Plain description of how customers pay. Per-seat, per-usage, tiered,
flat. Numbers live in `data/pricing.yaml` if a separate file exists;
this section names the *shape* of the model.

## Integrations

Canonical list of supported integrations. Update when a new integration
ships.

- Integration X (read/write)
- Integration Y (read-only)

## Known limitations

What buyers should know before signing. Honesty here saves churn later.

- Limitation 1
- Limitation 2
